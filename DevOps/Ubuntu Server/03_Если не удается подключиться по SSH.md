Если при первоначальном подключении вышла ошибка `Connection refused` , это означает, что ваш компьютер достучался до удаленного сервера, но на том конце "никто не взял трубку" — подключение на 22 порт было явно отклонено.

Поскольку сервер у вас "только что установленный", причина почти наверняка в нем самом. Вот пошаговый план, как это исправить (нужен доступ к самому серверу, например, через монитор и клавиатуру или консоль управления).

### 🎯 Шаг 1: Главная причина — не установлен SSH-сервер
На большинстве свежеустановленных Linux-систем (Ubuntu, Debian и др.) сервер SSH (`openssh-server`) **не установлен по умолчанию**. Без него ваш компьютер просто не с чем соединяться.

**Что делать на сервере:**
1. Откройте терминал.
2. Установите сервер:
   ```bash
   sudo apt update
   sudo apt install openssh-server -y
   ```
3. Проверьте, что он запустился:
   ```bash
   sudo systemctl status ssh
   ```
   В выводе должно быть зеленое `active (running)`. Чтобы выйти из просмотра статуса, нажмите `q`.

### ✅ Шаг 2: Убедитесь, что сервер "слушает" порт
Даже если служба установлена, она может быть не запущена или "слушать" нестандартный порт.

**Выполните на сервере:**
```bash
sudo ss -tlnp | grep :22
```
*   **Если вы видите** `LISTEN` и `0.0.0.0:22` — всё отлично, проблема в другом (см. Шаг 3).
*   **Если пусто** — служба не запущена. Запустите её: `sudo systemctl start ssh`, а затем добавьте в автозагрузку: `sudo systemctl enable ssh`.

### 🔥 Шаг 3: Проверьте и настройте брандмауэр (UFW)
Часто сервер установлен и работает, но подключение блокирует встроенный файрвол (UFW). Если вы его не настраивали специально, он может быть выключен, но проверить стоит.

**Выполните на сервере:**
```bash
sudo ufw status
```
Возможные варианты:
*   **Status: inactive** — брандмауэр выключен, пропускайте этот шаг.
*   **Status: active** — брандмауэр включен. Нужно явно разрешить SSH:
    ```bash
    sudo ufw allow ssh
    ```
    Или, если порт стандартный:
    ```bash
    sudo ufw allow 22/tcp
    ```

### 💡 Шаг 4: Особый случай — Виртуальная машина
Если вы используете виртуальную машину (VirtualBox, VMware) у себя на компьютере, обычная команда `ssh user@192.168.x.x` может не работать из-за настроек сети.
*   **Решение:** В настройках виртуальной машины переключите тип сетевого адаптера с **NAT** на **Bridged (Мост)**. После этого у виртуальной машины появится свой IP-адрес в сети, и подключение должно заработать.

### 🛠️ Если ничего не помогло: Диагностика
Включите подробный лог на вашем компьютере (не на сервере), чтобы понять, на каком этапе обрыв:
```bash
ssh -vvv ваше_имя_пользователя@192.168.3.6
```
*   Если видите `"Connection timed out"` — проблема в сети или файрволле по пути.
*   Если видите `"Connection refused"`, как сейчас, и все шаги выше выполнены — проверьте, не занят ли порт и точно ли запущен сервер командой `sudo systemctl status ssh`.

Попробуйте последовательно выполнить эти шаги на самом сервере. В большинстве случаев решение — просто установка пакета `openssh-server`. Напишите, на каком этапе возникнут трудности, разберемся дальше.
